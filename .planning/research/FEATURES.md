# Feature Landscape: Obsidian Note Generation from Telegram Bot

**Domain:** Telegram bot generating Obsidian-compatible markdown files via WebDAV to Yandex Disk
**Researched:** 2026-03-28
**Confidence:** MEDIUM-HIGH (official Obsidian docs + community sources + GitHub issues)

## Table Stakes

Features the generated notes MUST have, or they break in Obsidian or feel wrong to the user.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Valid YAML frontmatter block | Obsidian parses `---` delimited YAML at file top. Malformed YAML breaks Properties view and Dataview queries. | Low | Must be first thing in file, no leading whitespace |
| `tags` as YAML list (plural, no `#`) | Obsidian 1.9+ dropped support for `tag` (singular) and `cssclass`. Tags in frontmatter must NOT have `#` prefix. Must be a list, not a string. | Low | `tags:\n  - recipe\n  - food` not `tags: "#recipe"` |
| ISO 8601 dates in frontmatter | Obsidian and Dataview expect `YYYY-MM-DD` or `YYYY-MM-DDTHH:mm`. Other formats cause sort/query failures. | Low | `created: 2026-03-28` |
| Proper image embedding syntax | Images must use `![[filename.jpg]]` (Obsidian wiki-link) or `![alt](attachments/filename.jpg)` (standard markdown). | Low | Wiki-link is Obsidian-native; standard markdown is more portable. Recommend standard markdown for bot-generated notes. |
| Images stored in `attachments/` subfolder | Obsidian resolves images relative to note, then vault-wide. A single `attachments/` folder prevents name collisions and keeps vault clean. | Low | Must match Obsidian Settings > Files & Links > Default attachment folder |
| Sanitized filenames (no forbidden chars) | Obsidian forbids `* " \ / : \| ? [ ] # ^ \|` in filenames. Cyrillic is fine. | Med | Must sanitize title before using as filename. Replace forbidden chars with `-` or `_`. |
| UTF-8 encoding | Obsidian expects UTF-8. Cyrillic content and filenames must be properly encoded in WebDAV PUT. | Low | Node.js handles this natively |
| `.md` file extension | Obsidian only indexes `.md` files as notes. | Low | Trivial |
| Quoted YAML strings with special chars | Titles containing `:` or `'` break YAML parsing if unquoted. | Low | Always quote `title` values: `title: "Рецепт: Шарлотка"` |

## Obsidian Note Format Specification

### Minimal Well-Formed Note

```markdown
---
title: "Рецепт: Шарлотка с яблоками"
created: 2026-03-28
source: telegram
tags:
  - recipe
  - food
---

# Шарлотка с яблоками

Текст заметки здесь.

![photo](attachments/2026-03-28_sharlotka_abc123.jpg)
```

### Frontmatter Fields

**Natively recognized by Obsidian (special behavior):**

| Field | Type | Purpose | Required |
|-------|------|---------|----------|
| `tags` | list | Searchable tags, shown in Properties panel | Yes |
| `aliases` | list | Alternative note names for search/linking | No |
| `cssclasses` | list | Custom CSS styling per note | No |

**Custom fields (for Dataview queries and organization):**

| Field | Type | Purpose | Recommended |
|-------|------|---------|-------------|
| `title` | string | Note title (always quote) | Yes |
| `created` | date | ISO 8601 creation date | Yes |
| `source` | string | Origin: `telegram`, `forwarded`, `video` | Yes |
| `source_name` | string | Forwarded from channel/user name | If forwarded |
| `type` | string | Content type: `text`, `photo`, `video`, `forward` | Yes |
| `video_url` | string | YouTube/VK/RuTube link | If video |
| `category` | string | Mapped category folder name | Yes |

### YAML Formatting Rules

1. **Always use plural** property names: `tags` not `tag`, `aliases` not `alias`
2. **Always use list format** for tags, even with one tag:
   ```yaml
   tags:
     - recipe
   ```
   NOT: `tags: recipe`
3. **Always quote strings** containing `:`, `'`, `"`, `{`, `}`, `[`, `]`, or starting with special YAML characters
4. **Use lowercase keys** consistently: `created` not `Created`
5. **No nested YAML** -- keep frontmatter flat for Dataview compatibility
6. **No empty values** -- omit field rather than `field: ""`

### Image Embedding

**Use standard markdown syntax**, not Obsidian wiki-links:

```markdown
![описание](attachments/2026-03-28_sharlotka_abc123.jpg)
```

Rationale: Standard markdown syntax works in Obsidian AND in any other markdown viewer. Wiki-links (`![[file]]`) only work in Obsidian. Since notes are generated by a bot (not manually in Obsidian), standard syntax is safer and more portable.

**Image file naming convention:**
```
attachments/{date}_{sanitized_title}_{hash8}.jpg
```
Example: `attachments/2026-03-28_sharlotka_a1b2c3d4.jpg`

The 8-char hash suffix prevents collisions when two photos share a date and title.

### Tag Conventions

| Telegram Hashtag | Obsidian Tag | Notes |
|-----------------|--------------|-------|
| `#рецепт`, `#еда`, `#готовка` | `recipe` | Map to English for consistency |
| `#работа`, `#wb` | `work` | |
| `#идея`, `#проект` | `idea` | |
| Everything else | `misc` | Default |

Decision point: Russian vs English tags. English tags are safer for YAML (no quoting needed, no encoding edge cases) and work better with nested tags (`recipe/dessert`). But Russian is more natural for the user.

**Recommendation:** Use English tags in frontmatter. They are machine-generated metadata, not user-facing labels. The title and content remain in Russian.

## Filename Best Practices

### Format

```
{YYYY-MM-DD}_{sanitized_title}.md
```

Example: `2026-03-28_Шарлотка-с-яблоками.md`

### Sanitization Rules

1. Remove forbidden characters: `* " \ / : | ? [ ] # ^ |`
2. Replace spaces with `-`
3. Trim to max 80 characters (some filesystems have 255-byte limits; Cyrillic chars are 2 bytes in UTF-8)
4. Remove leading/trailing dots and spaces
5. Collapse consecutive `-` into one
6. If title is empty after sanitization, use `untitled`

### Why Date Prefix

- Natural chronological sort in file explorer
- Serves as deduplication namespace (same title on different days = different files)
- Matches the PROPFIND-based dedup strategy from PROJECT.md

## Folder Structure

```
vault/
  Рецепты/
    2026-03-28_Шарлотка.md
  Работа/
    2026-03-27_Отчёт-WB.md
  Идеи/
    2026-03-26_Парк-в-Дерябихе.md
  Разное/
    2026-03-25_Заметка.md
  attachments/
    2026-03-28_sharlotka_a1b2c3d4.jpg
```

**Single `attachments/` folder at vault root**, not per-category. Reasons:
- Simpler WebDAV operations (one MKCOL, one target path for all images)
- Obsidian resolves images vault-wide, so subfolder doesn't matter
- Prevents orphaned images when notes move between categories

## Remotely Save + WebDAV: How External Writes Work

### The Sync Model

Remotely Save compares local vs. remote files by **last-modified time**. When the bot writes a new `.md` file to Yandex Disk via WebDAV PUT:

1. File appears on Yandex Disk with a modification timestamp
2. Next time Obsidian opens (or auto-sync fires), Remotely Save runs PROPFIND
3. Plugin sees new remote file not present locally
4. File downloads to local vault
5. Obsidian indexes it, frontmatter parsed, note appears in search/tags/graph

**This is a one-way flow for bot-generated content.** The bot only creates new files. The user may edit them in Obsidian afterward, but the bot never overwrites existing notes (dedup by filename prevents this).

### Critical: Yandex Disk WebDAV Reliability Issues

**HIGH confidence -- multiple GitHub issues confirm this.**

Yandex Disk WebDAV has known problems with Remotely Save:
- **429 Too Many Requests** errors during sync (issues #869, #1026)
- Fix PRs (#1034, #1088) stalled due to inactive maintainer
- Files upload but don't download in some cases
- Yandex returns non-standard `202` responses to PUT (instead of `201`/`204`)

**Mitigations for bot-side writes:**
1. Add retry logic with exponential backoff for 429 responses
2. Add delay between consecutive WebDAV operations (don't batch-upload rapidly)
3. Verify upload with PROPFIND after PUT
4. The bot's write pattern (single note + maybe one image, sporadically) is low-volume and unlikely to trigger rate limits

**For Remotely Save sync side:**
- Consider the PRO "Yandex Disk native API" option instead of WebDAV for the Obsidian plugin sync
- Or accept that sync is "eventually consistent" and may need manual trigger

### What Remotely Save Does NOT Do

- Does not sync in background (only when Obsidian is open)
- Does not sync `.obsidian/` config folder by default
- Does not handle merge conflicts intelligently in free tier (just "keep newer" or "keep larger")
- Does not detect real-time changes -- polls on interval or manual trigger

### Encryption Consideration

If Remotely Save encryption is enabled, the bot CANNOT write plain markdown files that Obsidian can read. The bot writes plain `.md` files directly; Remotely Save would see them as unencrypted foreign files.

**Recommendation:** Do NOT enable Remotely Save encryption. The vault lives on Yandex Disk (user's own cloud), encryption adds complexity with zero benefit for single-user scenario.

## Differentiators

Features that would make the notes more useful but are not strictly required for basic functionality.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Auto-generated H1 heading from title | Obsidian shows filename in sidebar but H1 in reading view. Having both feels polished. | Low | Just add `# {title}` after frontmatter |
| `source_name` for forwarded messages | Knowing where content came from (which channel/person) is valuable for later recall | Low | Already parsed in existing bot code |
| Inline tags in body text | `#recipe` in body text lets user see tags while reading, not just in Properties | Low | Supplement to, not replacement for, frontmatter tags |
| `aliases` for transliterated title | Lets user find Russian note by typing Latin search | Med | Transliteration library needed |
| Video thumbnail as image | For video links, download and embed a thumbnail | Med | Requires yt-dlp or API call; adds visual richness |
| Backlinks via `[[related note]]` | Cross-referencing related notes leverages Obsidian's graph view | High | Requires knowing existing notes; overkill for MVP |

## Anti-Features

Features to explicitly NOT build.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Wiki-link image syntax `![[img]]` | Only works in Obsidian, breaks everywhere else | Use `![alt](path)` standard markdown |
| Nested/hierarchical frontmatter | Breaks Dataview queries, more complex to generate | Keep all fields flat |
| `#` prefix in YAML tags | Invalid in Obsidian 1.9+, causes parsing errors | Plain strings: `- recipe` not `- "#recipe"` |
| `tag` singular key | Deprecated in Obsidian 1.9+, will stop working | Always use `tags` (plural) |
| Storing videos in vault | Bloats sync enormously, Yandex Disk WebDAV is already fragile | Store video links in frontmatter `video_url` field |
| Remotely Save encryption | Bot writes plain files; encryption makes them unreadable by Remotely Save | Leave encryption off |
| Complex folder hierarchy | More MKCOL calls, harder hashtag-to-folder mapping, marginal benefit | 4-5 flat top-level folders max |
| Emoji in filenames | Can break on some sync services and filesystems | Stick to Cyrillic + Latin + numbers + hyphens |

## Feature Dependencies

```
Sanitized filenames → Deduplication via PROPFIND (filename must be predictable)
YAML frontmatter generation → Tag mapping (hashtags must map to clean tag strings)
Image upload to attachments/ → Image embedding in note body (path must match)
Folder creation (MKCOL) → Note upload (PUT) (folder must exist before file)
WebDAV connectivity → Everything (all storage operations depend on it)
```

## MVP Recommendation

### Must Have (Phase 1)

1. **YAML frontmatter generation** with `title`, `created`, `source`, `type`, `tags`, `category`
2. **Filename sanitization** with date prefix and Cyrillic support
3. **Hashtag-to-tag mapping** (existing hashtags to clean English tags)
4. **Hashtag-to-folder mapping** (existing logic, 4 folders + attachments)
5. **WebDAV PUT for .md files** with retry logic
6. **WebDAV PUT for images** to `attachments/` with proper naming
7. **Standard markdown image embedding** in note body
8. **MKCOL for folder creation** (idempotent, run on startup)
9. **PROPFIND deduplication** by filename

### Defer

- **Aliases/transliteration**: Nice but not needed for single user who knows their own content
- **Video thumbnails**: Extra API calls, extra complexity, links are sufficient
- **Backlinks**: Requires vault index, massive scope increase
- **Inline body tags**: Frontmatter tags are sufficient for search
